# Tempo — Gap Analysis & Roadmap

> เอกสารนี้ประเมินว่าแพลตฟอร์ม **Tempo (Agentic AI Entertain Assistant)** ตอบโจทย์กลุ่มเป้าหมายและปัญหาหลักของการซื้อ-จองตั๋วได้แค่ไหน เทียบกับ **โค้ดที่มีอยู่จริง** ณ วันที่ 30 พ.ค. 2026 พร้อมแผนพัฒนา (Roadmap) แบบเรียงลำดับก่อน-หลัง และประเมินความยาก
>
> สถานะ: 🟢 ใช้งานได้จริง · 🟡 มีบางส่วน/แค่แสดงผล · 🔴 ยังเป็น mock/ยังไม่มี

---

## 1. จุดยืนของแพลตฟอร์ม (Positioning)

ปัญหา 7 ข้อที่ตั้งไว้ **ส่วนใหญ่เป็นปัญหาของแพลตฟอร์มขายตั๋วเดิม** (ThaiTicketMajor, Ticketmelon, TheConcert) จุดยืนของ Tempo จึงมีได้ 2 ทาง:

| | **(A) ผู้ขายตั๋วเอง** | **(B) AI Agent ผู้ช่วย (แนะนำ)** |
|---|---|---|
| บทบาท | ขาย/จัดการ inventory เอง | ครอบบนแพลตฟอร์มเดิม ช่วยผู้ใช้ |
| ต้องสร้าง | payment, สต็อก, anti-bot, ส่งตั๋ว ครบเอง | ต่อคิวแทน, แจ้งเตือน, เทียบราคา, เฝ้าระวังตลาดรอง |
| ความเสี่ยง | แข่งตรงกับเจ้าตลาด + ภาระดำเนินงานสูง | พึ่งพา ToS/anti-bot ของแพลตฟอร์มปลายทาง |
| เหมาะกับ | ออร์แกไนเซอร์ที่อยากขายเอง | กลุ่มเป้าหมาย Gen Z/Y ที่อยาก "มีผู้ช่วย" |

**✅ ตัดสินใจแล้ว (2026-05-30):** Tempo เป็น **ตัวกลางการสั่งซื้อตั๋ว (intermediary/broker)** ที่มี **Agentic AI คอยแนะนำผู้ใช้** — กล่าวคือ ผู้ใช้สั่งซื้อ/ชำระเงินผ่าน Tempo (Tempo เป็นช่องทางสั่งซื้อ) แต่ AI ทำหน้าที่ค้นหา เปรียบเทียบ และแนะนำตัวเลือกที่ดีที่สุดให้ ไม่ใช่ปล่อยให้ผู้ใช้หาเอง เหมือน "ผู้ช่วยส่วนตัว + ช่องทางจอง" ในที่เดียว

### 1.1 คุณค่าหลัก: ตั๋ว + ที่พักใกล้ venue (Concert + Stay)
กลุ่มเป้าหมายสำคัญคือ **คนเดินทางไกลมาดูคอนเสิร์ต** เมื่อจองตั๋วแล้ว มักต้องหาโรงแรม/คอนโดใกล้สถานที่จัดงานเพื่อเดินทางสะดวก-รวดเร็ว Tempo จึงต้องเสนอ **แพ็กเกจ "ตั๋ว + ที่พัก"** ในขั้นตอนเดียว:
- หลังเลือกตั๋ว → AI แนะนำโรงแรม/คอนโดใกล้ venue อัตโนมัติ (เรียงตามระยะทาง/ราคา)
- รองรับทั้ง **โรงแรม** และ **คอนโด/ที่พักรายวัน** (ตรงกับแท็บ Stay ใน Admin)
- เชื่อม `event.venue` → คำนวณ `distance_to_venue` → จองที่พักต่อเนื่องจากตั๋ว
- โมเดลรายได้เสริม: affiliate (Agoda/Booking/Klook) ที่มีช่องตั้งค่าไว้แล้ว

## 2. กลุ่มเป้าหมาย vs สิ่งที่มี

| กลุ่มเป้าหมาย | ความต้องการหลัก | สถานะแพลตฟอร์ม |
|---|---|---|
| Gen Z (คุ้นเทคโนโลยี) | UX ลื่น, AI ช่วยตัดสินใจ, มือถือ | 🟢 AI chat + UI โมเดิร์น + voice input |
| Gen Y (นักศึกษา/วัยทำงานตอนต้น) | จบในที่เดียว: ตั๋ว+โรงแรม+ปฏิทิน | 🟡 มีครบทุกหน้า แต่ booking ยัง mock |
| แฟน K-Pop / T-Pop | เตือนรอบขาย, ต่อคิว, กันพลาด | 🔴 ยังไม่มี alert/queue จริง |
| นักท่องเที่ยวสาย Event | ที่พักใกล้ venue, แพ็กเกจ | 🟡 มี Hotels/Condos + `distance_to_venue` แต่จองไม่ได้จริง |

**สรุป:** ดีไซน์ตรงกลุ่มเป้าหมายดีมาก จุดที่ยังขาดคือ "เครื่องยนต์" ที่ทำงานจริงเบื้องหลัง

## 3. ปัญหา 7 ข้อ vs โค้ดจริง (Gap Analysis)

| # | ปัญหา | แพลตฟอร์มควรช่วยอย่างไร | สถานะ | ช่องว่างหลัก |
|---|---|---|:---:|---|
| 1 | ระบบล่ม / เด้งออกหน้าจ่ายเงิน | จ่ายเงินทนทาน, retry ได้, สถานะชัด | 🔴 | จ่ายเงินเป็น modal จำลอง ไม่มี state machine/idempotency |
| 2 | ตัดเงินแล้วไม่ได้ตั๋ว | reconcile ทุกธุรกรรม, คืนเงินอัตโนมัติ | 🔴 | ไม่มี webhook / การกระทบยอด / กู้คืนรายการ |
| 3 | บอทกวาดตั๋ว + โกงตลาดรอง | กันบอท, resale ที่ verify, โอนตั๋วปลอดภัย | 🔴 | CAPTCHA ปลอม client-side, ไม่มี resale/transfer |
| 4 | ราคาไม่โปร่งใส | แยกค่าธรรมเนียมชัดก่อนจ่าย | 🟡 | แสดงราคา/โซน ✅ แต่ไม่มี fee breakdown |
| 5 | ความปลอดภัย / ป้องกันบอท | rate-limit, device check, CAPTCHA จริง | 🔴 | `middleware.ts` ไม่มี throttle, key 2Captcha/Browserbase ไม่ถูกใช้ |
| 6 | ข้อสัญญา / เลื่อน-ยกเลิก / สมาชิกพิเศษ | flow คืนเงิน, แจ้งเลื่อนงาน, pre-sale | 🔴 | มี field `cancelled` แต่ไม่มี flow ใด ๆ |
| 7 | ส่งตั๋วช้า + support แย่ | e-ticket จริง, QR/Wallet, support ticket | 🔴 | QR/อีเมลเป็นภาพจำลอง, support = อีเมล+FAQ |

### สิ่งที่ "ทำงานได้จริง" แล้ว (ของดีที่มี)
- 🟢 **Auth** (Supabase OAuth + magic link)
- 🟢 **Event browse / Bookings list / Notifications** (อ่าน-เขียน Supabase จริง)
- 🟢 **Scrapers** TTM/Ticketmelon/TheConcert/Agoda/Booking (Playwright + LLM) — แต่ trigger เองจาก Admin เท่านั้น ยังไม่มี cron
- 🟢 **AI Chat** ต่อ LLM จริง (OpenRouter/Ollama) — แต่ "ลงมือจอง/ต่อคิว" ยังเป็น mock
- 🟢 **FAQ**

## 4. Roadmap (เรียงลำดับก่อน-หลัง)

ประเมินความยาก: ⭐ ง่าย · ⭐⭐ กลาง · ⭐⭐⭐ ยาก · ⭐⭐⭐⭐ ยากมาก (ต้องพึ่งภายนอก/ลงทุนสูง)

### Phase 0 — ความซื่อสัตย์ของผลิตภัณฑ์ (ทำทันที) ⭐ ✅ เสร็จแล้ว
แก้ก่อนเพราะกระทบความน่าเชื่อถือ และทำให้เดโมไม่เข้าใจผิด
- [x] ลดคำที่ "อ้างเกินจริง" ใน UI → "CAPTCHA Bypass" เป็น "CAPTCHA Assist", footer/welcome แชตใส่ป้าย **โหมดสาธิต**, feature flags + e-ticket/calendar/wallet ใส่ **(เร็ว ๆ นี้)**
- [ ] เคาะจุดยืน A/B ให้ชัด ← **ยังรอผู้ใช้ตัดสินใจ**
- **ผล:** ป้องกันความเข้าใจผิด, พร้อมโชว์ investor/ผู้ใช้อย่างตรงไปตรงมา

### Phase 1 — หัวใจการทำธุรกรรม (ลำดับสูงสุด) ⭐⭐⭐ 🟡 วางรากฐานแล้ว
แก้ปัญหา **#1, #2, #4** ซึ่งเป็นเหตุผลหลักที่คนเลิกใช้แพลตฟอร์มเดิม
- [x] **Payment foundation** — `lib/server/payments.ts` + `app/api/payments/*` (Omise PromptPay หลัก, Stripe optional, **demo provider** เดิน flow จริงได้โดยไม่ตัดเงิน)
- [x] **Idempotency key** — กดซ้ำ/ระบบล่มแล้วลองใหม่ ใช้ key เดิม → ไม่ตัดเงินซ้ำ
- [x] **Webhook + reconciliation** — `reconcile()` ถามสถานะจริงจาก provider ทุกครั้งที่ poll → แก้เคส "จ่ายแล้วไม่ได้ตั๋ว"
- [x] **Price breakdown** — แยกราคาบัตร + ค่าบริการ + ยอดรวม ในหน้าจ่ายเงิน
- [ ] **ยังเหลือก่อน production:** ตาราง `payments` ใน DB (ตอนนี้เก็บใน memory), การ์ดจริงผ่าน Omise.js/Stripe.js token, verify signature ของ webhook, สมัคร+KYC Omise/Stripe
- **ผล:** จองผ่าน flow จริงได้ (เดโม) + ปัญหา "ตัดเงินแต่ไม่ได้ตั๋ว" แก้เชิงสถาปัตยกรรมแล้ว

### Phase 2 — การส่งมอบตั๋ว + แพ็กเกจที่พัก ⭐⭐⭐
แก้ปัญหา **#7 (ส่วนการส่งตั๋ว)** + สร้างคุณค่าหลัก "ตั๋ว + ที่พักใกล้ venue" (ดู §1.1)
- [ ] **QR/E-ticket จริง** (encode `ticket_code` + ลายเซ็น กันปลอม)
- [ ] **ส่งอีเมล** (Resend/SendGrid) แนบ e-ticket
- [ ] **เพิ่มลงปฏิทิน** จริง (ไฟล์ `.ics` ก่อน, Google Calendar API ทีหลัง)
- [ ] **Bundle ตั๋ว + ที่พัก** — หลังจองตั๋ว AI แนะนำโรงแรม/คอนโดใกล้ venue → จองต่อเนื่องในขั้นตอนเดียว, รวมยอด, ใช้ payment foundation เดิม
- [ ] **จองที่พักจริง** ผ่าน affiliate/partner (โรงแรม + คอนโด) — ต่อยอดจากแท็บ Stay ที่มีปุ่ม โรงแรม/คอนโด แล้ว
- **ผล:** ผู้ใช้ได้ตั๋วในมือ + ที่พักใกล้งาน จบในแอปเดียว (จุดต่างหลักจากแพลตฟอร์มตั๋วทั่วไป)

### Phase 3 — ความปลอดภัย & กันบอท ⭐⭐⭐
แก้ปัญหา **#3 (ฝั่งกันบอท), #5**
- [ ] **Rate limiting** ใน `middleware.ts` (เช่น Upstash) ต่อ IP/ผู้ใช้
- [ ] **CAPTCHA จริง** (Cloudflare Turnstile/hCaptcha) ตอน login/checkout
- [ ] (ถ้าเดินทาง B) ต่อ **Browserbase/2Captcha** สำหรับ auto-queue จริง + human-in-the-loop
- **ผล:** ลดบอท, ระบบทนต่อโหลดช่วงเปิดขาย

### Phase 4 — ตลาดรอง & สิทธิ์ ⭐⭐⭐⭐
แก้ปัญหา **#3 (ฝั่งตลาดรอง), #6**
- [ ] **Verified resale / transfer** — โอนตั๋วผ่านระบบ (ออก QR ใหม่ ป้องกันขายซ้ำ), เพดานราคา
- [ ] **Refund / cancellation / postponement** flow + แจ้งเตือนอัตโนมัติ
- [ ] **Pre-sale / membership** สิทธิ์เข้าก่อน
- **ผล:** ปิดช่องโกงตลาดรอง + จัดการเลื่อน/ยกเลิกเป็นระบบ

### Phase 5 — บริการลูกค้า & ระบบอัตโนมัติ ⭐⭐
แก้ปัญหา **#7 (ฝั่ง support)**
- [ ] **Support ticket / live chat** (หรือ AI assistant ตอบ + ส่งต่อมนุษย์)
- [ ] **Cron auto-sync** สำหรับ scrapers (ตอนนี้ trigger มือ)
- [ ] **Alert รอบเปิดขาย** (push/อีเมล) สำหรับแฟน K-Pop/T-Pop
- **ผล:** ครบ loop ประสบการณ์ผู้ใช้

---

## 5. ลำดับแนะนำให้ลงมือ

```
Phase 0 (ตอนนี้) → Phase 1 Payment (หัวใจ) → Phase 2 ส่งตั๋ว → Phase 3 ความปลอดภัย → Phase 4/5
```

**เหตุผล:** ไม่มีประโยชน์ที่จะกันบอท (Phase 3) ถ้ายัง "จองจริงไม่ได้" (Phase 1) — ต้องมีธุรกรรมจริงก่อน แล้วค่อยทำให้ปลอดภัยและขยายผล

---

## 6. ความเสี่ยง/ข้อพึ่งพาภายนอกที่ต้องรู้
- **Payment:** ต้องสมัคร Omise/Stripe + ผ่าน KYC ธุรกิจ (ใช้เวลา) — ระหว่างนี้ใช้ "demo provider ที่เดิน flow จริง" ทดสอบได้
- **Scrapers/auto-queue:** Playwright รันบน Vercel serverless ไม่ได้ (ใหญ่/ต้อง long-running) — ต้องมี worker host แยก
- **Anti-bot ของแพลตฟอร์มปลายทาง:** ถ้าเดินทาง B การ auto-queue อาจชน ToS — ต้องตรวจสอบกฎหมาย/ข้อตกลง
